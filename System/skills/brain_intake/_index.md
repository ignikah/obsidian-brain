# LLM-Wiki（/intake）

> Wiki 是永久複利物件。每次寫入都讓整個知識庫更緊密；每次查詢都可能產出新的知識頁面。
> 人只管輸入來源和發問，CKO 處理所有整合與維護。

**觸發**：`/intake`、`/wiki`、「歸檔」、「整理」、「我想問...」

---

## 三個核心操作

### 1. Ingest — 寫入即整合

**觸發情境**：使用者丟入任何來源（URL、檔案、筆記、口述、貼文）

**流程**：

1. **讀懂來源** — 提取核心概念、實體、決策、洞察（不只是摘要）
2. **判斷 PARA 歸屬** — 確定目標路徑，添加 frontmatter
3. **識別關聯頁面** — 在現有知識庫找 3–15 篇相關筆記（依內容複雜度調整數量）
4. **跨頁面整合** — 逐一更新關聯頁面：
   - 補充新資訊到對應段落
   - 加入互連引用 `[[新頁面]]`
   - 發現矛盾則在兩頁面都標記 `> ⚠️ 矛盾待確認：`
5. **建立或更新目標頁面** — 寫入新知識，不覆蓋既有內容，而是整合
6. **更新 `_index.md`** — 在目標目錄的成員清單加入新頁面
7. **迴圈檢查** — L3 frontmatter → L2 成員清單 → L1 分類體系

**輸出報告**（每次 ingest 後回報）：
```
已寫入：{目標路徑}
已整合：{N} 篇相關頁面
  - {頁面A}：補充了 {內容}
  - {頁面B}：加入互連引用
矛盾標記：{N} 處（需人工確認）
```

---

### 2. Query — 問答即知識

**觸發情境**：使用者對知識庫提問

**流程**：

1. **識別相關頁面** — 根據問題找出知識庫中的相關條目
2. **綜合回答** — 整合多頁面內容，附上引用來源 `[[頁面名]]`
3. **評估回存價值** — 若回答本身包含新洞察或值得複用的推論，提議回存：
   - 「這個分析要存入知識庫嗎？建議路徑：`3_Resources/{domain}/`」

**輸出格式**：
```
（回答內容）

來源：[[頁面A]]、[[頁面B]]
---
建議回存：是 / 否
```

---

### 3. Lint — 內容品質健康檢查

**觸發情境**：`/wiki lint`、「檢查知識庫品質」

> 與 `/stale` 的分工：`/stale` 偵測**結構性**腐敗（孤島、空洞、斷裂）；Lint 偵測**內容性**腐敗。

**檢查項目**：

| 類型 | 說明 | 偵測方式 |
|------|------|---------|
| 矛盾 | 兩頁面對同一事實陳述不同 | 搜尋 `⚠️ 矛盾` 標記 + CKO 主動比對 |
| 過時聲明 | 引用的外部資訊可能已更新 | 找有 URL 來源且超過 180 天未更新的頁面 |
| 孤立頁面 | 無其他頁面引用的知識條目 | 無 `[[反向引用]]` 的頁面 |
| 缺漏交叉引用 | 概念上相關但未互連 | CKO 閱讀後人工判斷 |

**輸出**：一份 `System/working-memory/wiki-lint-YYYY-MM-DD.md` 報告，格式同 `/stale` 報告。

---

## 設計原則

- **知識複利**：每次 ingest 不只新增一頁，而是讓 N 頁同時變得更完整
- **人策劃，AI 維護**：使用者輸入來源和提問；CKO 處理所有交叉引用、矛盾標記、成員清單同步
- **回答即知識**：有價值的 Query 回答可以直接回存為新頁面，避免洞察消散在對話歷史裡
